Built-in Functions vs. UDFs

This section explores how you apply row-level logic. Standard Python UDFs (User Defined Functions) are the #1 performance killer in PySpark because they force data out of the JVM into a Python process, serializing row by row. Always prefer Spark's native C-optimized built-in functions.

Execution Time Comparison

Avoid: Python UDF
from pyspark.sql.functions import udf
from pyspark.sql.types import IntegerType

# Defines a standard Python function
def square(x):
    if x is not None:
        return x * x
    return None

# Registers it as a UDF (Forces serialization)
square_udf = udf(square, IntegerType())

# Extremely slow on large datasets
df = df.withColumn("squared", square_udf(df.value))
Prefer: Spark Built-ins
import pyspark.sql.functions as F

# Executes directly in the JVM via Catalyst
# Leverages Tungsten physical execution

df = df.withColumn("squared", 
                   F.col("value") * F.col("value"))

# OR using mathematical built-ins
# df = df.withColumn("squared", F.pow(F.col("value"), 2))

Optimizing Joins & Shuffles

Joining tables is the most common cause of network bottlenecks (shuffles). This section explains how to use Broadcast joins to eliminate network traffic when combining a massive table with a smaller lookup table.

The Broadcast Hash Join

If one DataFrame is small enough to fit in the memory of a single executor (typically < 10MB to 2GB depending on config), you should "broadcast" it. Spark sends a full copy of the small table to every executor, completely eliminating the need to shuffle the large table across the network.

Join Strategy Implementation

Sort-Merge Join (Default): Both tables are hashed, shuffled across the network so matching keys land on the same node, sorted, and then merged. High network I/O.

large_transactions_df = spark.read.parquet("...")
small_stores_df = spark.read.parquet("...")

# Triggers a massive network shuffle of transactions
joined_df = large_transactions_df.join(
    small_stores_df,
    on="store_id",
    how="inner"
)

Broadcast Hash Join: The small table is shipped to all nodes. The large table is read and joined locally without ever moving across the network. Extremely fast.

import pyspark.sql.functions as F

large_transactions_df = spark.read.parquet("...")
small_stores_df = spark.read.parquet("...")

# The 'broadcast' hint instructs Spark to ship the small table
joined_df = large_transactions_df.join(
    F.broadcast(small_stores_df),
    on="store_id",
    how="inner"
)

Partitions: Repartition vs Coalesce

Controlling the number of partitions dictates your cluster's parallelism. Too few partitions means idle CPUs. Too many means overhead. This section covers how to safely change partition counts.

Network IO Comparison

Decreasing partitions from 100 to 10

🔥 df.repartition(n)

Does a full network shuffle of the data to create equal-sized partitions.

  • Use to increase partitions.
  • Use to fix severely skewed data (using a column: repartition("col")).
  • Expensive! Involves full data movement.

⚡ df.coalesce(n)

Combines existing partitions on the same node. Minimizes data movement.

  • Use only to decrease partitions.
  • Use before writing to disk to avoid 10,000 tiny files.
  • Cheap! Avoids full network shuffles.

Memory Management: Caching

Spark uses lazy evaluation. If you derive a complex DataFrame and use it in multiple downstream actions (like writing to multiple places, or displaying and saving), Spark will recompute it from scratch every time unless you cache it.

The Caching Lifecycle

# 1. Read Raw Data raw_df = spark.read.parquet("s3://data/raw/") # 2. Expensive Transformations (Joins, Aggregations, Regex) clean_df = raw_df.join(ref_df, "id").groupBy("category").sum("amount") # 3. CACHE the intermediate result # Use .persist() if you need to specify MEMORY_ONLY, MEMORY_AND_DISK, etc. clean_df.cache() # 4. First Action: Triggers computation AND caches data in executors total_count = clean_df.count() # 5. Second Action: Lightning fast. Reads directly from memory! clean_df.write.format("delta").save("s3://data/gold/") # 6. UNPERSIST: Crucial step to free up cluster memory when done clean_df.unpersist()
